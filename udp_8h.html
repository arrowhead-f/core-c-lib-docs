udp.h File Reference

Detailed Description

User Datagram Protocol (UDP) utilities.

The UDP protocol facilitates unreliable transmission of messages, or datagrams, over IP networks. These datagrams are both sent and received using sockets. To give you more control over the medium through which these sockets communicate, we also provide a mechanism we refer to as transports. Please refer to RFC8085 to learn more about UDP itself. Below, we briefly describe how to use this C API.

Note
When we use the terms remote and local throughout this file, we do so from the perspective of individual sockets rather than complete devices. In other words, when we consider a certain socket, that socket is local and whatever other socket it may communicate with is remote.

Sockets

To set up a local socket, you call ah_udp_sock_init() and ah_udp_sock_open(), in that order. Successfully initialized sockets are terminated with ah_udp_sock_term() and successfully opened sockets are closed with ah_udp_sock_close(). If you wish to configure a socket by setting any of its options, such as by using ah_udp_sock_set_reuseaddr(), it is typically most appropriate to do so after it has been opened and before it is used to send or receive data. Every socket may receive datagrams, and is notified of other events, via a socket observer of type ah_udp_sock_obs.

As UDP is a connection-less communication protocol, opening a socket is enough for both sending and receiving datagrams. To receive datagrams from any other UDP socket, you must first enable it via a call to ah_udp_sock_recv_start(). You may disable receiving again with a call to ah_udp_sock_recv_stop(). To send datagrams, use ah_udp_sock_send().

Transports

Most functions you can invoke in this header immediately delegate to a transport, represented by the ah_udp_trans type, rather than dealing with the invocation itself. When you first initialize a socket with ah_udp_sock_init(), you specify what transport to use with that particular socket. This arrangement means that functionality can be injected into code relying on the API of this header.

Most uses of this API will likely involve the default transport, which you can get instances of via ah_udp_trans_get_default(). This transport engages the actual UDP networking capabilities of the underlying platform. You could, however, use a custom transport for testing purposes. Or you may use a transport that logs sent and received messages and then passes on any calls it receives to another transport.

◆ ah_udp_in_alloc_for()

ah_extern ah_err_t ah_udp_in_alloc_for ( ah_udp_in_t **  owner_ptr)

Allocates new input buffer, storing a pointer to it in owner_ptr.

The allocated input buffer is stored to owner_ptr and contains its own copy of owner_ptr. The buffer can later be detached from its owner by a call to ah_udp_in_detach(), which sets the copy to NULL and replaces the pointer pointed to by owner_ptr with that of a new input buffer.

Every input buffer allocated with this function must eventually be provided to ah_udp_in_free(). It is the responsibility of the owner of each instance to make sure this is the case.

Parameters
owner_ptrPointer to own pointer to allocated input buffer.
Returns
One of the following error codes:
  • AH_ENONE - The operation was successful.
  • AH_EINVAL - owner_ptr is NULL.
  • AH_ENOMEM - No enough heap memory available (ah_palloc() returned NULL).
  • AH_EOVERFLOW - AH_PSIZE is too small for it to be possible to store both an ah_udp_in instance and have room for input data in a single page provided by the page allocator (see ah_palloc()).
Note
This function should primarily be of interest to those both wishing to implement their own UDP transports and need to intercept buffers (for the sake of decryption, for example).

◆ ah_udp_out_alloc()

ah_extern ah_udp_out_t * ah_udp_out_alloc ( void  )

Dynamically allocates and partially initializes a UDP output buffer.

The size of the payload memory of the returned buffer

Every output buffer allocated with this function must eventually be provided to ah_udp_out_free().

Concretely, the page allocator (see ah_palloc()) is used to allocate the returned buffer. All parts of the returned buffer are initialized, except for the actual payload memory.

Returns
Pointer to new output buffer, or NULL if the allocation failed.
Warning
If AH_PSIZE is configured to a too small value (see conf.h), this function always fails.

Schedules closing of sock.

If the return value of this function is AH_ENONE, meaning that the closing could indeed be scheduled, its result will eventually be presented via the ah_udp_sock_cbs::on_close callback of sock.

Parameters
sockPointer to socket.
Returns

  • AH_EINVAL - sock is NULL or the transport of sock is missing a required function in its virtual function table.
  • Any additional code returned by the used UDP transport.

This option is an important way of preventing multicast messages from leaving the local network of the current device. For example, on Linux, the default multicast hop limit is 1. Generally, the value should be small.
